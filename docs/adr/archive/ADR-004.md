# ADR-004: Implementação das Recomendações da Ultra-Auditoria v2.0.2

## Status
Implementado (Fase A + Fase B + Fase C concluídas)

## Contexto

A Ultra-Auditoria do repositório `ignite-agents-skills` v2.0.2 (executada em 2026-07-05) aplicou a rubrica `skill-audit-bulletin` ao pacote inteiro e produziu um veredito **B — 81.5/100** ("ADOTAR COM AJUSTES"). A auditoria identificou 9 débitos técnicos concretos (§4) e propôs 6 skills novas para fechar lacunas estruturais do SDLC (§6).

### Débitos Identificados (§4 do relatório)

| ID | Débito | Severidade | Esforço |
|----|--------|------------|---------|
| §4.1 | Sobreposição `planning` vs `writing-plans` — "Não use quando" quase idêntico | Média | S |
| §4.2 | Drift de versão — zip é 2.0.2, `index.json` diz 2.0.1 | Baixa | S |
| §4.3 | CHANGELOG parado no mundo pré-refactor — zero registro do salto v2.0.x | Média | S |
| §4.4 | CI valida estrutura, não qualidade — `validate-skill.sh` existe mas não está no pipeline | Alta | S |
| §4.5 | Divergência da convenção `agentskills.io` — `templates/` vs `scripts/` | Baixa | Decisão |
| §4.6 | Colisão de nome `architecture-review` com skill em outro ecossistema | Média | M |
| §4.7 | `skill-audit-bulletin` órfã no grafo de `related_skills` | Baixa | S |
| §4.8 | Duas skills conceituais correm risco de padding para bater 150 linhas | Baixa | S |
| §4.9 | `docs/skill-maintenance.md` descreve processo de equipe que não existe | Baixa | S |

### Skills Novas Propostas (§6 do relatório)

| Prioridade | Skill | Lacuna que fecha | Relevância atual |
|------------|-------|-------------------|------------------|
| 🥇 #1 | `security-review` | Nenhuma skill ensina revisão de segurança | App Secure Notes em produção (AES-256-GCM, TOTP, scrypt, BIP39) |
| 🥈 #2 | `agent-orchestration` | Nenhuma skill cobre coordenação multi-agente | Stack Kilo com 9 agent-model assignments |
| 🥉 #3 | `data-modeling` | `ddd` modela domínio, não schema físico | App imobiliário (schema relacional) |
| #4 | `api-design` | Contraparte de git/release/testing inexistente | backends Express (Secure Notes, imobiliário) |
| #5 | `observability` | Nenhuma skill cobre instrumentação | Operador solo sem time olhando dashboard |
| #6 | `refactoring` | `architecture-review` diagnostica, ninguém prescreve tratamento | Fecha ciclo diagnóstico→tratamento |

## Decisão

Implementar as recomendações da Ultra-Auditoria em duas fases:

### Fase A: Correção de Débitos Técnicos (7 tarefas, ~3-4h)
Tarefas rápidas, baixo risco, paralelizáveis. Fecham os buracos que mais vão mendar em 3-6 meses.

### Fase B: Skills Novas (6 skills, ~12-16h)
Skills que seguem o padrão Ultra-High Quality Grade existente (≥150 linhas, decision tree, workflows com checkpoints, anti-patterns com severidade, templates, cross-references).

### Decisão sobre §4.5 (Divergência de convenção)
Manter a estrutura `templates/`, `examples/`, `checklists/` como Kilo-first. A compatibilidade `agentskills.io` citada no README é aspiracional, não um compromisso imediato. Reavaliar quando o custo de migração for justificável (≥20 skills ou demanda externa real).

### Decisão sobre §4.6 (Colisão `architecture-review`)
Renomear a skill deste repo para `architecture-review-kilo` (mais nova, mais específica). A skill externa (`/mnt/skills/user/architecture-review/`) mantém o nome original.

## Alternativas Consideradas

### Alternativa A: Ignorar débitos e focar só em skills novas
- **Prós**: Mais rápido, entrega valor novo imediato
- **Cons**: Débitos acumulados degradam confiança no sistema de validação

### Alternativa B: Implementar tudo de uma vez (big bang)
- **Prós**: Consistência garantida, tudo entrega junto
- **Cons**: Alto risco de retrabalho, difícil de revisar

### Alternativa C: Fase A + Fase B sequencial (escolhida)
- **Prós**: Débitos resolvidos primeiro (base sólida), skills novas beneficiam do CI corrigido
- **Cons**: Depende de disciplina de execução

## Consequências

### Positivas
- CI valida qualidade, não só estrutura (fecha §4.4)
- CHANGELOG reflete a história real do projeto (resolve §4.2 e §4.3)
- Grafo de `related_skills` fica conexo (resolve §4.7)
- 6 novas skills fecham lacunas reais do SDLC
- Total de 20 skills "Ultra-High Quality Grade"

### Negativas
- ~15-20h de trabalho adicional
- `architecture-review-kilo` pode confusar quem conhece a skill externa
- Manutenção de 20 skills (vs 14) aumenta custo operacional

### Riscos
- **Risco**: Skills novas ficam "thin" para bater 150 linhas
  - **Mitigação**: Usar `validate-skill.sh` como gate + revisão editorial extra em `security-review` e `agent-orchestration`
- **Risco**: Renomear `architecture-review` quebra referências existentes
  - **Mitigação**: Buscar todas as ocorrências antes de renomear, atualizar `index.json` e todos os `related_skills`

# ADR-010: Branch Protection e SemVer para Tags — Emergencial

## Status
Implementado (emergencial)

## Data
2026-07-05

## Contexto

Durante a implementação de ADRs, foi identificado que:
1. Mudanças são feitas diretamente em `master` sem branch de trabalho isolado
2. Tags de release são reaproveitadas (ex: v2.1.0 foi recriada)
3. Falta padronização de quando criar branches e como versionar releases

## Decisão

**Regra obrigatória:** Sempre que uma skill de implementação de ADR for invocada (feat, fix, refactor, etc), deve-se:

1. **Criar branch de trabalho adequada:**
   - `feature/` para novas funcionalidades
   - `fix/` para correções
   - `docs/` para documentação
   - `adr-XXX/` para implementação de ADR específica

2. **Fluxo obrigatório:**
   ```
   Branch de trabalho → Implementação → Validação 100% → Merge → gh-pages sync → Tag SemVer
   ```

3. **SemVer para tags:**
   - **NUNCA reaproveitar tags**
   - Sempre criar nova tag após modificação
   - Tags seguem `vMAJOR.MINOR.PATCH`
   - Releases são feitas após merge na master

## Consequências

### Positivas
- Isolamento de mudanças
- Histórico limpo
- Versionamento previsível
- GitHub Pages sempre sincronizado

### Negativas
- Mais passos no fluxo

### Riscos
- Nenhum significativo

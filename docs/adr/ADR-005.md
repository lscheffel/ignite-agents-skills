# ADR-005: Introdução da Skill `implementation` para Execução Governada de Mudanças

## Status

Proposto

## Contexto

O ecossistema de Agent Skills implementa com maturidade as fases de análise, documentação e planejamento do SDLC, porém apresenta uma lacuna estrutural entre a geração dos artefatos de decisão e sua efetiva execução.

### Fluxo Atual (com lacuna)

```
architecture-review
        │
        ▼
adr-generator
        │
        ▼
planning
        │
        ▼
writing-plans
        │
        ▼
?????  ← LACUNA
        │
        ▼
testing
        │
        ▼
git
        │
        ▼
release
```

### Diagnóstico

| Capacidade | Status | Skills Responsáveis |
|------------|--------|---------------------|
| Diagnosticar código | ✅ | `architecture-review-kilo`, `ddd` |
| Decidir mudanças | ✅ | `adr-generator` |
| Planejar implementação | ✅ | `planning`, `writing-plans` |
| **Executar mudanças** | ❌ | **Nenhuma** |
| Testar mudanças | ✅ | `testing` |
| Versionar mudanças | ✅ | `git` |
| Publicar mudanças | ✅ | `release` |
| Documentar mudanças | ✅ | `documentation` |

### Consequências da Lacuna

- Cada agente executa mudanças segundo heurísticas próprias
- Não existe contrato de execução
- Não existe validação formal entre ADR, Blueprint e TODO
- Não existe sequência padronizada para implementação
- Agentes frequentemente iniciam alterações sem verificar consistência entre artefatos
- Tarefas podem ser executadas em ordem incorreta
- Documentação, testes e governança podem ser esquecidos durante a implementação

## Decisão

Introduzir uma nova skill denominada **`implementation`**, responsável exclusivamente pela execução governada de mudanças previamente planejadas.

### Princípio Fundamental

A skill **não** produzirá arquitetura.

A skill **não** produzirá planejamento.

A skill **não** substituirá ADRs.

A skill **consumirá** os artefatos produzidos pelas demais skills e coordenará sua implementação.

### Responsabilidades

| Responsabilidade | Descrição |
|------------------|-----------|
| Interpretar ADRs | Extrair contexto, decisão e consequências |
| Interpretar Blueprints | Extrair estrutura, dependências e estimativas |
| Interpretar TODOs | Extrair tarefas, prioridades e critérios |
| Validar consistência | Verificar coerência entre os três artefatos |
| Construir plano interno | Montar DAG de execução a partir do TODO |
| Executar alterações | Implementar incrementalmente com validação |
| Validar estado | Build, lint, test, typecheck após cada ciclo |
| Atualizar documentação | Sincronizar ADRs, Blueprint, TODO, docs |
| Atualizar progresso | Manter estados das tarefas no TODO |
| Produzir relatório | Gerar Execution Report ao término |

### Posição no SDLC

```
architecture-review
        │
        ▼
adr-generator
        │
        ▼
planning
        │
        ▼

        │
        ▼

        │
        ▼

```

## Alternativas Consideradas

### Alternativa A: Continuar com prompts livres
- **Prós**: Zero implementação
- **Cons**: Execução inconsistente, variável entre agentes, sem governança

### Alternativa B: Expandir `writing-plans`
- **Prós**: Menor número de skills
- **Cons**: Mistura planejamento com execução, viola responsabilidade única

### Alternativa C: Skill dedicada `implementation` (Escolhida)
- **Prós**: Separação clara, execução reproduzível, governança explícita
- **Cons**: Mais uma skill para manter, acoplamento com diversas outras

## Consequências

### Positivas
- Fecha o ciclo completo do SDLC baseado em Agent Skills
- Reduz comportamento heurístico dos agentes
- Aumenta previsibilidade e repetibilidade
- Padroniza implementações entre diferentes modelos
- Melhora rastreabilidade e facilita auditorias
- Cria pipeline totalmente dirigido por artefatos (Artifact-Driven SDLC)

### Negativas
- Skill de elevada complexidade (8 áreas funcionais)
- Exige manutenção contínua
- Forte acoplamento com diversas outras skills

### Riscos
- **Risco**: Skill pode ser interpretada como "gerador de código"
  - **Mitigação**: Nome `implementation` (não `code-generator`), escopo limitado a mudanças já planejadas
- **Risco**: Acoplamento com muitas skills pode gerar fragilidade
  - **Mitigação**: Contratos de I/O bem definidos, dependências declaradas no frontmatter

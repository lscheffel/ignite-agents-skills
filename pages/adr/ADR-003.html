Decisões Tomadas

1. Template Base Único

Decisão: Usar um template base (skill-template.md) para todas as skills.

Rationale: Garante consistência estrutural e permite validação automatiza. O template define seções obrigatórias (Quando Usar, Workflow, Anti-patterns, Checklists, Edge Cases) e opcionais.

Trade-off: Algumas skills conceituais (vibe-coding, prompt-engineering) precisaram de adaptações para não forçar estrutura rigidamente.

2. Anti-patterns com Severidade

Decisão: Usar indicadores visuais (🔴🟡🟢) para severidade de anti-patterns.

Rationale: Permite que agentes priorizem evitar erros críticos primeiro. Formato visual é rápido de escanear.

Trade-off: Pode parecer informal para documentação técnica, mas a experiência mostra que é altamente eficaz para agentes.

3. Decision Trees com Mermaid

Decisão: Usar grafos Mermaid para decision trees.

Rationale: Grafos são mais fáceis de seguir que listas textuais. Mermaid é suportado pelo GitHub e pode ser renderizado.

Trade-off: Agentes precisam de habilidade para interpretar grafos, mas a maioria dos LLMs modernos consegue.

4. Templates Externos vs Inline

Decisão: Mover templates para arquivos separados (.md, .yml, .ts).

Rationale: Permite reuso independente, versionamento separado, e agentes podem baixar apenas o que precisam.

Trade-off: Mais arquivos para manter, mas o benefício de reuso compensa.

5. Skill Nova: skill-audit-bulletin

Decisão: Criar uma skill para auditar outras skills.

Rationale: Fornece processo sistemático para avaliar qualidade de skills, útil para manutenção e adoção de novas skills.

Trade-off: Mais uma skill para manter, mas resolve problema real de validação.

O que Funcionou

  1. Processo Sistemático: Seguir o template base garantiu consistência
  2. Validação Automatizada: O script validate-index.sh pegou erros antes do commit
  3. Incremental: Refatorar skill por skill permitiu learning contínuo
  4. Cross-references: Criar rede de conhecimento entre skills aumentou utilidade
  5. Examples Reais: Exemplos antes/depois mostram valor concreto

O que Não Funcionou

  1. Tempo Estimado: Subestimamos esforço — skills complexas (DDD) tomaram mais tempo
  2. Templates Complexos: Alguns templates ficaram grandes demais para uso rápido
  3. GitHub Pages: Deploy manual causou inconsistências temporárias entre branches
  4. Branching Strategy: Falta de estratégia clara causou conflitos entre master e gh-pages

Lições Aprendidas

  1. Documentar Antes de Implementar: O blueprint (ADR-002-BP) salvou horas de retrabalho
  2. Validação Automatizada é Essencial: Sem o script, erros passariam despercebidos
  3. Incremental > Big Bang: Refatorar uma skill por vez é mais seguro
  4. Templates Devem Ser Simples: Complexidade demais reduz adoção
  5. GitHub Pages Requer Branch Dedicada: Não misturar desenvolvimento e deploy

Impacto

Positivo

  • Skills agora são instruções acionáveis, não apenas documentação
  • Agentes podem seguir workflows deterministicamente
  • Templates permitem reuso consistente
  • Decision trees habilitam raciocínio automático
  • Checklists permitem validação de qualidade

Negativo

  • Mais arquivos para manter (40 templates, 13 examples)
  • Curva de aprendizado para novos contribuidores
  • Algumas skills podem ficar "pesadas" demais para uso rápido

Próximos Passos

  1. Testar com Agentes Reais: Validar que agentes conseguem seguir as skills
  2. Coletar Feedback: Usar skill-audit-bulletin para avaliar eficácia
  3. Iterar: Ajustar templates baseado em uso real
  4. Documentar Padrões: Criar guia de estilos para templates

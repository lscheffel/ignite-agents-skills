ADR-006: Workflow CI para Auto-sync do Index e Deploy GitHub Pages

Status

Implementado

Contexto

O repositório ignite-agents-skills mantém um registry centralizado de skills em skills/index.json, que serve como índice para o GitHub Pages hospedado na branch gh-pages. Atualmente, o processo de manutenção deste index é totalmente manual:

Fluxo Atual (com lacuna)

Desenvolvedor cria nova skill
        │
        ▼
Desenvolvedor roda ./scripts/sync-index.sh manualmente
        │
        ▼
Desenvolvedor commita mudanças no index.json
        │
        ▼
Desenvolvedor faz push para master
        │
        ▼
????? ← LACUNA (deploy manual para gh-pages)
        │
        ▼
GitHub Pages desatualizada

Diagnóstico

CapacidadeStatusEvidence
Criar skills✅Template + workflow documentado
Validar skills✅validate-skill.sh + CI workflow
Sincronizar index⚠️Script existe, mas é manual
Deploy GitHub Pages❌Nenhum workflow automático
Sync master → gh-pages❌Nenhum workflow automático

Consequências da Lacuna

  • index.json pode ficar desatualizado quando novas skills são adicionadas
  • GitHub Pages pode estar inconsistentes com o estado atual do repo
  • Processo manual é propenso a esquecimentos
  • Contributors precisam lembrar de rodar sync-index.sh antes de commitar
  • Não há garantia de que index está sincronizado antes do merge

Decisão

Criar um workflow de CI/CD automatizado que:

  1. Detecta mudanças na pasta skills/ (adição, remoção ou modificação de skills)
  2. Sincroniza automaticamente skills/index.json usando scripts/sync-index.sh
  3. Valida o index sincronizado usando scripts/validate-index.sh
  4. Commita a mudança no index.json na branch master
  5. Faz push da branch master atualizada
  6. Deploya automaticamente para a branch gh-pages via GitHub Actions

Arquitetura do Workflow

Push para master (com mudanças em skills/)
        │
        ▼
┌─────────────────────────────────────┐
│  GitHub Actions Workflow            │
│                                     │
│  1. Checkout (com token GITHUB_TOKEN)│
│  2. Setup (node, jq)               │
│  3. Executar sync-index.sh         │
│  4. Executar validate-index.sh     │
│  5. Git add + commit (se houver     │
│     mudanças no index.json)         │
│  6. Git push para master            │
│  7. Checkout gh-pages              │
│  8. Merge master → gh-pages        │
│  9. Git push gh-pages              │
└─────────────────────────────────────┘
        │
        ▼
GitHub Pages atualizada automaticamente

Detalhes da Implementação

Trigger: push para branch master com paths skills/**

Jobs:

  1. sync-and-deploy
  • Checkout com fetch-depth: 0 e token GITHUB_TOKEN
  • Executar sync-index.sh
  • Executar validate-index.sh
  • Configurar git (user.name, user.email)
  • Verificar se index.json mudou (git diff --quiet)
  • Se mudou: commit + push para master
  • Checkout gh-pages
  • Merge master (com strategy ours para conflitos)
  • Push para gh-pages

Segurança:

  • GITHUB_TOKEN já tem permissão de write via permissions: contents: write
  • Workflow só roda em push para master (não em PRs)
  • Validação impede deploy de index quebrado

Idempotência:

  • Se index.json já está sincronizado, nenhum commit é gerado
  • Se gh-pages já está atualizado, nenhum push é feito

Alternativas Consideradas

Alternativa A: Deploy via GitHub Pages (branch selection)

  • Prós: Configuração mais simples
  • Cons: Não sincroniza index automaticamente; requer merge manual

Alternativa B: Deploy estático com GitHub Actions (upload-pages-artifact)

  • Prós: Mais controle sobre o que é deployado
  • Cons: Complexidade desnecessária; já temos branch gh-pages funcionando

Alternativa C: Workflow completo com sync + deploy (Escolhida)

  • Prós: Totalmente automatizado; zero intervenção manual
  • Cons: Workflow mais complexo; requer permissões de write no repo

Consequências

Positivas

  • index.json sempre sincronizado com o estado atual das skills
  • GitHub Pages sempre reflete o estado mais recente do repo
  • Zero intervenção manual necessária
  • Contributors não precisam lembrar de rodar sync-index.sh
  • Validação automática impede deploy de index quebrado
  • Processo é auditável via logs do GitHub Actions

Negativas

  • Workflow adiciona complexidade ao pipeline de CI
  • Commit automático pode poluir histórico (mitigado com mensagem clara)
  • Requer permissões de write no repo (já disponível via GITHUB_TOKEN)
  • Conflitos de merge são possíveis (mitigados com strategy ours)

Riscos

  • Risco: Workflow pode falhar por falta de permissão
  • Mitigação: permissions: contents: write já configurado
  • Risco: Conflitos de merge entre master e gh-pages
  • Mitigação: Strategy ours assume que master é a fonte de verdade
  • Risco: Loop infinito se workflow commita e dispara outro run
  • Mitigação: Commits automáticos via actions/checkout@v4 não disparam workflows (por padrão)

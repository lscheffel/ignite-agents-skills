Skill Audit Bulletin

Audita skills existentes avaliando qualidade, completude, acionabilidade e risco.

Quando Usar

Use quando:

  • Precisa revisar uma skill existente
  • Precisa validar registry de skills
  • Precisa decidir adoção de skill
  • Precisa comparar skills similares
  • Precisa auditoria de qualidade

Não use quando:

  • Skill é nova (use skill-template)
  • Skill é trivial (não precisa auditoria)
  • Apenas leitura (sem decisão)

Skills relacionadas:

  • architecture-review-kilo — para revisão de código
  • documentation — para padrões de documentação

Workflow

Fase 1: Auditoria Estrutural

  1. Verifique frontmatter:
   head -10 skills/{skill}/SKILL.md
  1. Valide campos obrigatórios:
  • name, description, version, tags, related_skills
  1. Execute script de validação:
   bash scripts/validate-skill.sh skills/{skill}
  1. Checkpoint: Frontmatter válido, estrutura correta

Fase 2: Análise Semântica

  1. Analise description para keywords:
  • Palavras-chave de gatilho
  • Lacunas de cobertura
  1. Verifique decision tree:
  • Clareza dos caminhos
  • Exemplos incluídos
  1. Checkpoint: Triggering preciso, fronteira clara

Fase 3: Avaliação de Workflow

  1. Conte fases de workflow:
   grep -c "^### Fase" skills/{skill}/SKILL.md
  1. Verifique checkpoints:
   grep -c "Checkpoint" skills/{skill}/SKILL.md
  1. Checkpoint: Workflows completos com validação

Fase 4: Correção Técnica

  1. Valide exemplos:
   # Teste comandos em shell
   # Verifique código compila
  1. Verifique anti-patterns:
  • Severidade (🔴🟡🟢)
  • Exemplos antes/depois
  1. Checkpoint: Conteúdo técnico correto

Fase 5: Gerar Bulletin

  1. Use template templates/audit-bulletin.md
  2. Preencha scores:
  • Triggering: 0-10
  • Aplicabilidade: 0-10
  • Profundidade: 0-10
  • Correção: 0-10
  • Universalidade: 0-10
  • Manutenibilidade: 0-10
  • Ergonomia: 0-10
  • Risco: 0-10 (invertido)
  1. Checkpoint: Bulletin completo com veredito

Conceitos Fundamentais

Anti-patterns

🔴 Crítico

Audit sem Scoring Objetivo

O que é: Auditoria sem scores numéricos. Por que é ruim: Decisão subjetiva, impossível comparar. Como evitar: Use weights e scores padronizados. Exemplo:

# ❌ ERRADO
"Skill está boa"

# ✅ CORRETO
"Skill: 85/100 (B) — ADOTAR COM AJUSTES"

Falso Veredito

O que é: Recomendação que não corresponde ao score. Por que é ruim: Confusão, decisões erradas. Como evitar: Veredito deve seguir score. Exemplo:

# ❌ ERRADO
Score 45/100, mas "ADOTAR AS-IS"

# ✅ CORRETO
Score 45/100 → "REFATORAR"

🟡 Médio

Audit superficial

O que é: Auditoria que só verifica estrutura. Por que é ruim: Problemas técnicos não detectados. Como evitar: Teste exemplos, verifique código. Exemplo:

# ❌ ERRADO
"Frontmatter OK, skill aprovada"

# ✅ CORRETO
"Frontmatter OK, exemplos testados, workflow validado"

🟢 Baixo

Audit sem Data

O que é: Bulletin sem data de auditoria. Por que é ruim: Impossível rastrear histórico. Como evitar: Sempre inclua data. Exemplo:

# ✅ CORRETO
**Audit date:** 2024-01-15

Edge Cases

Skill sem Templates

Situação: Skill sem pasta templates/. Solução: Score reduzido em 10-15%. Exceção: Skills conceituais podem não precisar.

Skill sem Examples

Situação: Skill sem exemplos práticos. Solução: Score reduzido em 5-10%. Exceção: Skills teóricas.

Skill com Conflito

Situação: Skill sobrescreve outra. Solução: Marcar como risco alto. Exceção: Se é intencional, documentar.
